# SOA2 — Attacker-Percentage Sensitivity Analysis (SCBC / VCBC)

**Baseline:** Alabdulatif, Alharbi, Mchergui, Moulahi (2024),
*"Mitigating Blackhole and Greyhole Routing Attacks in VANETs Using Blockchain
Based Smart Contracts"*, CMES vol.138 no.2, pp.2005–2021.
DOI 10.32604/cmes.2023.029769.

**Implementation:** SCBC (Algorithms 1–3) and VCBC (Algorithms 4–5) deployed as
Go chaincode on a real Hyperledger Fabric network (see `README.md`).

---

## 1. Purpose of this experiment

Implementing the state-of-the-art method alone is not sufficient: we must show
how its performance *responds* to the experiment's independent variable. For a
greyhole/blackhole defence the natural independent variable is the **attacker
percentage** — the fraction of vehicles in the network that are malicious.

This report sweeps the attacker percentage from **5 % to 60 %** and measures
four metrics for both smart contracts:

| Metric | Definition | Paper Eq. |
|--------|------------|-----------|
| Classification accuracy | (TP + TN) / N | §5.2 |
| TPR (recall) / FPR | TP/(TP+FN), FP/(FP+TN) | §5.2 |
| Network PDR | delivered relays / total relays | Eq. 1 |
| Routing overhead | (D_net + D_ctrl) / D_net | Eq. 3 |

**Experiment configuration.** N = 30 vehicles per network; 30 random seeds per
attacker level (curves show mean ± standard deviation); malicious nodes split
evenly between blackholes (drop ≈ everything) and greyholes (partial dropping
near the classification threshold τ = 50 %). Generated by `scbcvcbc_sweep.py`;
raw data in `results/soa2_sweep_results.csv`.

**Modelling note (why the curves are not flat).** On a small test network a
naïve attacker model produces flat, uninformative lines, because synthetic
blackhole/greyhole behaviour is trivially separable from honest behaviour. To
obtain a realistic response we model **downstream-path contamination**: a node's
*observed* forwarding ratio reflects its own behaviour **and** the malicious
relays further along its path. As attacker density rises, honest nodes' observed
PDR sags toward the threshold and greyholes straddle it — the regime where the
metrics genuinely move.

## 3. Detection accuracy vs attacker percentage

![Detection accuracy](../../results/soa2_sweep_acc.png)

*Figure 2 — Classification accuracy of SCBC and VCBC as the attacker fraction
increases.*

**Reading the figure.**

- **SCBC stays high (≈ 0.98 → 1.00).** With no prior knowledge it classifies
  purely on the delivered ratio; because it never excludes nodes pre-emptively
  it produces **zero false positives** (FPR = 0 throughout), so its only errors
  are missed greyholes at very low attacker counts.
- **VCBC sits lower (≈ 0.89 → 0.96).** Its miner-voting pre-filter (Alg. 4)
  occasionally votes an honest node grey/black and excludes it, which is counted
  as a false positive. This reproduces the paper's central finding that VCBC's
  voting layer trades a little precision.
- Both curves **improve as attacker density rises**, because heavily-dropping
  nodes become statistically easier to separate.

| Attacker % | SCBC acc | VCBC acc |
|-----------:|---------:|---------:|
| 5  | 0.983 | 0.891 |
| 20 | 0.982 | 0.901 |
| 40 | 0.994 | 0.924 |
| 60 | 1.000 | 0.956 |

## 4. Network packet delivery ratio vs attacker percentage

![Network PDR](../../results/soa2_sweep_pdr.png)

*Figure 3 — Network PDR (Eq. 1) as the attacker fraction increases.*

**This is the headline trend.** Network PDR falls **monotonically from ≈ 0.91 at
5 % attackers to ≈ 0.71 at 60 % attackers** — a clear, smooth response to the
independent variable. SCBC and VCBC track each other closely here: once a node is
classified white and used as a relay, its delivery behaviour is the same under
either contract, so the difference between the methods is in *who they exclude*,
not in the PDR of the surviving relays.

| Attacker % | Network PDR |
|-----------:|------------:|
| 5  | 0.91 |
| 20 | 0.85 |
| 40 | 0.78 |
| 60 | 0.71 |

---

## 5. TPR / FPR vs attacker percentage

*(Bottom-left panel of Figure 1.)*

- **TPR rises with attacker density** (SCBC 0.75 → 1.00; VCBC 0.80 → 1.00).
  At low density, greyholes whose dropping is close to the threshold are
  occasionally missed; as contamination grows their observed PDR sags below τ
  and they are reliably caught.
- **FPR:** SCBC ≈ 0 % at every level; VCBC ≈ 9–12 %, driven entirely by the
  noisy voting pre-filter. This is the precision cost of VCBC's earlier
  decisions.

---

## 6. Routing overhead vs attacker percentage

![Routing overhead](../../results/soa2_sweep_ro.png)

*Figure 4 — Routing overhead (Eq. 3) vs attacker percentage.*

Routing overhead is **flat (≈ 1.01)** across the sweep. This is expected and
correct: Eq. 3 charges one ~100-byte smart-contract control message per node,
so the overhead scales with **network size, not attacker mix**. Holding N fixed,
the attacker percentage does not change it. The flat line is itself a useful
result — it shows the defence's control cost does not blow up as the network
becomes more hostile.

---

## 7. Summary

- The metrics respond meaningfully to the independent variable: **PDR degrades
  linearly with attacker percentage**, detection **TPR improves** as attacks
  intensify, and **routing overhead stays constant**.
- **SCBC** maximises accuracy and avoids false positives but reacts only after
  observing behaviour. **VCBC** acts earlier through miner voting at the cost of
  a ~10 % false-positive rate — exactly the SCBC-vs-VCBC trade-off reported in
  the original paper, now reproduced on a real blockchain deployment.
- **Caveat.** These PDRs come from a parametric VANET model (with path
  contamination), not directly from the NS-3 packet-level run. The same trend is
  reproducible through the live chaincode (`scbcvcbc_sweep.py --backend fabric`);
  the local backend is used only to average enough seeds for smooth curves. A
  follow-on would drive the sweep from NS-3 simulations at different
  `--attack_number` / attacker-count settings, feeding each run's
  `vcbc_detection.csv` through the existing bridge.
